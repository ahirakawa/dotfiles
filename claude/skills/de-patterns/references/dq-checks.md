# Data Quality チェックの実践パターン

Source: https://www.startdataengineering.com/post/types-of-dq-checks/

## チェックの優先順位

**Output → Input → Mid-pipeline の順に実装する。**

全部入りにしない。コストとランタイムのバランスが重要。

## Output DQ チェック（最優先）

下流の消費者を不正データから守る。

### 1. テーブル制約チェック
- NOT NULL、UNIQUE、値域チェック
- 例: 「売上は返金以外は常に正」

### 2. ビジネスルールチェック
- ドメイン固有のロジック検証
- 例: 「購入日 > 商品公開日（予約販売を除く）」

### 3. スキーマチェック
- カラム名・型がラン間で一貫しているか
- 型の暗黙キャストが入り込んでいないか

### 4. メトリクス変動チェック
- KPI の急激な変動を検知（前日比、ローリング標準偏差）
- 罠: 季節性を考慮しないと誤検知が頻発する

### 5. 行数分布チェック
- ラン間で行数が急増/急減していないか

### 6. 入出力突合チェック
- 入力行数と出力行数の整合性
- JOIN の粒度ミスマッチによる行数膨張を検知

## Input DQ チェック（第2優先）

上流の問題を早期に捕捉。

- テーブル制約チェック（Output と同様）
- スキーマ検証
- 参照整合性チェック（JOIN キーの存在確認）
- 行数の一貫性

**スキップしてよい場合**: パイプラインが安く高速に実行できる場合

## Mid-Pipeline チェック

複雑な多段階変換では、各ステップ後にバリデーション。
根本原因の特定を高速化し、再実行範囲を最小化。

## Audit ロギング

各ステージの行数を記録して、行数を歪める問題（重複生成JOIN、誤ったフィルタ）を特定。

## 鮮度モニタリング（バックグラウンドジョブ）

パイプラインとは別に実行:
- `現在時刻 - MAX(insert_timestamp)` でデータ鮮度を監視
- パイプラインをブロックしないが、遅延を検知

## 重大度レベル

| レベル | 内容 | アクション |
|--------|------|-----------|
| Low | パターン監視、軽微な行数変動 | ログ記録のみ |
| Medium | 欠損データ、値域違反 | 定期的に対応 |
| High | スキーマ違反、ビジネスルール違反 | パイプライン即時停止 |

## よくある実装ミス

| ミス | 何が起きるか |
|------|------------|
| チェックの入れすぎ | パイプラインのコストとランタイムが膨張 |
| 季節性の無視 | メトリクス変動チェックで誤検知が頻発 |
| Audit trail の省略 | 行数異常の根本原因が特定できない |
| 1テスト=1スキャン | テーブルスキャンがチェック数分倍増 → コスト爆発 |
